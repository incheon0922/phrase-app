# 成语词汇积累小程序

一个面向言语板块备考场景的词汇积累与刷题小程序，核心目标是帮助用户沉淀成语、近义词、易错词和释义，并基于已积累词库自动生成练习题，形成“导入 -> 整理 -> 练习 -> 复盘”的闭环。

当前项目基于 `Taro + React + TypeScript`，优先面向微信小程序开发。

## 1. 项目目标

本项目主要解决以下问题：

1. 用户已有大量词汇积累内容，分散在 `Word`、`TXT`、笔记中，整理成本高。
2. 词汇积累后缺少高频练习入口，记忆不牢、复习效率低。
3. 练习结果无法沉淀，做错的词、易混淆的词无法持续强化。

因此，本项目会围绕以下核心能力设计：

1. 支持导入已有词汇资料。
2. 自动解析词条并形成结构化词库。
3. 基于词库自动生成四选一练习题。
4. 记录练习结果，支持错题回看和复习强化。

## 2. 核心需求拆解

### 2.1 词汇导入

支持用户导入已有积累内容，目标格式包括：

1. `TXT`
2. `Word`（`.docx` 优先，`.doc` 可作为兼容目标）

建议的导入方式分两期实现：

#### MVP 方案

1. 支持 `TXT` 文件导入。
2. 支持粘贴文本导入。
3. 支持手动新增词条。

原因：

1. 微信小程序对本地文件解析能力有限。
2. `Word` 直接在小程序端解析实现复杂，兼容性差。
3. 先把词库结构、练习链路和数据闭环跑通，性价比最高。

#### 推荐正式方案

1. 小程序端选择文件并上传。
2. 服务端或云函数解析 `docx/txt`。
3. 将非结构化文本转换为标准词条。
4. 返回解析结果供用户确认后入库。

推荐理由：

1. 更适合处理 `Word` 文档。
2. 后续更容易扩展 `PDF`、`Excel`、网页剪藏。
3. 可统一做格式清洗、重复去重、异常提示。

### 2.2 首页练习

首页进入后弹出或展示“开始练习”入口，点击后进入练习流程。

题目规则建议如下：

1. 每次展示 1 个题目。
2. 题干可优先采用“成语 -> 释义匹配”模式。
3. 提供 `A/B/C/D` 四个选项。
4. 选对后当前选项变绿。
5. 选错后当前选项变红，并高亮正确答案为绿。
6. 答题后展示简单解析，支持进入下一题。

优先支持的题型：

1. 根据成语选释义。
2. 根据释义选成语。

第二阶段可扩展的题型：

1. 近义词辨析。
2. 反义词辨析。
3. 语境填空。
4. 易错字判断。

### 2.3 底部导航

底部 `TabBar` 包含：

1. 首页
2. 我的

页面职责如下：

#### 首页

1. 展示开始练习入口。
2. 展示今日练习数据摘要。
3. 展示最近导入词汇数、待复习数、错题数。

#### 我的

当前按需求先保留：

1. 授权栏
2. 退出登录

建议同步补充：

1. 个人信息展示
2. 导入记录
3. 错题本入口
4. 清空本地数据
5. 关于与隐私说明

## 6. 数据模型建议

### 6.1 词条结构 `VocabularyItem`

```ts
export interface VocabularyItem {
  id: string
  word: string
  meaning: string
  example?: string
  synonym?: string[]
  antonym?: string[]
  tags?: string[]
  source?: 'txt' | 'docx' | 'manual'
  createdAt: string
  updatedAt: string
  masteryLevel: number
  wrongCount: number
  favorite: boolean
}
```

说明：

1. `masteryLevel` 用于控制复习频率。
2. `wrongCount` 用于错题强化。
3. `tags` 可区分成语、近义词、易错词等。

### 6.2 题目结构 `PracticeQuestion`

```ts
export interface PracticeQuestion {
  id: string
  type: 'word_to_meaning' | 'meaning_to_word'
  stem: string
  options: string[]
  answer: string
  explanation?: string
  vocabularyId: string
}
```

### 6.3 作答记录 `PracticeRecord`

```ts
export interface PracticeRecord {
  id: string
  questionId: string
  vocabularyId: string
  selectedOption: string
  isCorrect: boolean
  answeredAt: string
}
```

## 7. 导入解析规则建议

为了提升导入成功率，建议支持以下文本格式。

### 7.1 纯文本格式

```txt
按部就班：按照一定的步骤、顺序进行
白驹过隙：形容时间过得很快
不刊之论：指不能改动或不可磨灭的言论
```

解析规则：

1. 按行分割。
2. 用 `：`、`:`、`-`、空格等作为候选分隔符。
3. 左侧识别为词语，右侧识别为释义。

### 7.2 分段文本格式

```txt
词语：按部就班
释义：按照一定的步骤、顺序进行
例句：备考要按部就班，不可急于求成
```

适用于更完整的词条解析。

### 7.3 Word 导入策略

建议优先支持 `.docx`，因为其本质是结构化压缩包，更容易在服务端解析。

推荐流程：

1. 小程序上传文件。
2. 服务端提取段落文本。
3. 调用统一解析器转成词条数组。
4. 返回前端预览确认。

## 8. 出题逻辑建议

### 8.1 基础出题流程

1. 从词库随机抽取一个目标词条。
2. 根据题型生成题干。
3. 从词库中抽取 3 个干扰项。
4. 打乱顺序后生成 `A/B/C/D` 选项。

### 8.2 干扰项生成原则

建议优先选择相近词作为干扰项，这样训练价值更高：

1. 词义相近
2. 字形相似
3. 同类标签
4. 历史高频混淆词

### 8.3 作答反馈

1. 选对：当前选项绿色。
2. 选错：所选项红色，正确项绿色。
3. 展示题目解析。
4. 更新当前词条的掌握状态。

## 9. 技术方案建议

## 9.1 前端技术栈

当前项目已采用：

1. `Taro`
2. `React`
3. `TypeScript`
4. `Sass`

适合原因：

1. 一套代码可兼容微信小程序和 H5。
2. 组件化开发更利于后续扩展页面。
3. TypeScript 更适合处理词条、题目、记录等结构化数据。

### 9.2 状态管理

首期建议优先使用轻量方案：

1. 页面局部状态使用 React Hooks。
2. 全局数据可先封装 `store` 或 `context`。
3. 数据持久化使用小程序本地缓存。

当功能复杂后再考虑引入更完整的状态管理方案。

### 9.3 数据存储

建议分两层：

#### 本地存储

适用于：

1. 词库缓存
2. 练习记录
3. 登录态缓存

#### 云端存储

适用于：

1. 多端同步
2. Word 文件解析
3. 用户账号隔离
4. 数据备份

推荐路径：

1. 首期仅本地存储完成 MVP。
2. 二期接入微信云开发或独立后端。

## 12. 关键交互细节

### 12.1 答题反馈

1. 点击选项后立即显示正确/错误状态。
2. 已答题状态下不可重复提交。
3. 可增加 `600ms - 1000ms` 的短暂停顿，再进入下一题，体验更自然。

### 12.2 空状态处理

当用户尚未导入词库时，首页应提示：

1. 暂无词汇数据
2. 去导入词汇
3. 或手动新增第一条词汇

### 12.3 导入失败处理

应明确提示：

1. 文件格式不支持
2. 文件内容为空
3. 无法识别有效词条
4. 存在重复词条

## 13. 风险与注意事项

### 13.1 Word 解析复杂度

`Word` 解析并不适合直接在小程序前端硬做，推荐服务端处理。否则会遇到：

1. 文件读取限制
2. 解析库兼容性问题
3. 包体积增大
4. 复杂格式识别失败

### 13.2 干扰项质量决定题目质量

如果 3 个错误选项过于随意，练习价值会明显下降，所以后续最好维护：

1. 同标签词汇池
2. 相似义项池
3. 高频混淆词对

### 13.3 数据结构要先定

即使首期只是本地存储，也建议先把词条、题目、记录结构定义稳定，否则后续导入、统计、同步都会返工。
